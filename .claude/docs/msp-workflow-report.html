00 Bottom line up front the one idea + the diagnosis

The single most important idea

“Merged into the shared branch” and “visible to users” are two separate events. Every parallel-safe technique exploits that gap: code lives on the shared branch, the build stays green, teammates are never blocked — but users see nothing until a final small commit flips it on. The discipline holding it together is tiny, reversible commits to one mainline, verified by automated tests before each merge. High triangulated across martinfowler.com, DORA, trunkbaseddevelopment.com

MERGED

The gap — incomplete work lives here, safely
Feature FlagsKeystone InterfaceDark LaunchingBranch by AbstractionExpand / Contract
RELEASED
visible / active for users · the final small commit turns it on

The five techniques in the middle column all do one job: keep unfinished work mergeable and non-breaking, so “deploy” can happen continuously while “release” waits for the right moment.

Headline diagnosis of your current config

Your config already nails the shape of a good unit of work (atomic commits, ~200–400 LOC, “commit often / perfect later”, squash-on-merge). The gaps are about timing, reach, and model: (1) commits are rule-locked to “only when the user asks”; (2) work stays local — git push is not auto-allowed and nothing pushes between waves; (3) plan execution validates only once at the very end, so mid-plan states are not provably shippable; and (4) the branching rule models only “feature branch off default” — it has zero coverage of trunk-direct work, a shared development integration branch, feature flags, or expand-contract.

Net: excellent for one developer building one branch; thin for continuous, parallel-safe, multi-developer shipping.

01 What is an “MSP”? the unit of work

An MSP (Minimum Shippable Product), as you frame it, is the smallest change that can be independently built, tested, and deployed to production on its own. Empirically that means completable in hours to a couple of days, merged to a shared mainline at least once every 24 hours. High

Terminology honesty

“MSP” is your term — it is not in the DORA / Fowler / Humble canon. It maps cleanly onto DORA’s “small batch” plus the “thin vertical slice”. Read the rest of this report as synthesizing across vocabularies, not quoting a single named standard. High

02 Branching & continuous integration where work merges

Plain definitions. Trunk / mainline = the one shared branch (main / master). Integration = merging your work with everyone else’s. Continuous Integration (CI) = every developer merges to mainline at least daily and an automated build + test runs immediately. CI is a human-frequency discipline, not just “having a build server”. High Fowler 2024

Trunk-Based Development (TBD): all devs collaborate on one trunk, either committing directly (small experienced teams) or via short-lived branches lasting no more than ~2 days. DORA defines TBD as: merge to trunk at least daily, branches last hours, ≤3 active branches, no code freezes. Scale proof: Google runs 35,000 devs on a single monorepo trunk. High trunkbaseddevelopment.com + dora.dev

Your master → development → feature maps closely to a GitLab-Flow / three-branch variant: master (production), development (integration), feature/* (unit of work). Direct commits to development are safe only if: (1) CI runs on every push to development; (2) master is protected and only receives deliberate release merges from development; (3) the commit is genuinely small and non-breaking (docs, config, a one-liner). Anything touching business logic / APIs / shared state should still get a short-lived feature branch + PR back into development, regardless of size — so review happens before integration and development stays green.

Recommendation: three-branch variant + CI gates on development + GitHub-Flow discipline (every substantive change via PR, branch deleted on merge) to stop feature branches aging into merge hazards. High DORA + GitHub Flow + GitLab Flow docs

Objectivity — TBD’s weakness

DORA explicitly notes TBD fails without strong automated tests and a synchronous code-review culture. Long-lived feature branches also suppress refactoring (Fowler, FeatureBranch): devs avoid refactoring shared code to dodge merge conflicts, so the codebase rots. High

03 Shipping without breaking others parallel-safe techniques

These techniques decouple “merged / deployed” from “released / active” so incomplete work lives safely on the shared branch. This directly answers pain (c): how plan output can stay individually mergeable without breaking parallel developers.

Plain definitions. Toggle / flag = runtime on/off switch. Consumer = any code / service / user calling an API or reading a column. Abstraction layer = a thin wrapper hiding the real implementation. Green build = all mainline tests pass. Toggle debt = accumulated stale flags. Idempotent migration = safe to run more than once.

1 · Feature Flags

An if/else switch read at runtime; both code paths live on mainline; flip it on when ready, no redeploy. The cheapest way to merge unfinished work.

Cost: toggle debt (Knight Capital, 2012), test-matrix growth (2^N states), cleanup discipline. High

2 · Keystone Interface

Build all backend across releases; add the single user-facing UI entry point in one final small commit. Until then the feature is invisible — no flag needed.

Prefer over a release toggle when there’s one clean UI chokepoint. Needs backend testable without the UI. High

3 · Dark Launching

Call new backend logic from real production traffic but discard the result; tests performance and correctness at production scale with zero user-visible change.

Best for mandatory system processes (e.g. every checkout). Differs from canary (canary shows the feature to N% of real users). High

4 · Branch by Abstraction

Insert a thin wrapper between callers and the old module, route through it, build the new module behind it, migrate callers gradually, then delete old + wrapper. Whole replacement in small mainline commits, always green.

Use for big structural replacements (ORM, payment lib). Cost: temporary dual implementation. High

5 · Parallel Change / Expand-Contract

Change an interface or schema in 3 phases: Expand (add new alongside old) → Migrate (move callers one at a time) → Contract (remove old). Each phase a separate reversible migration.

Aligns with your no-direct-DB rule: each phase is a SQL file the human applies; additive-first keeps the DB compatible with both old and new code. Cost: discipline to finish Contract. High

04 Commits & pull requests the mechanics

Plain definitions. Commit = a referenceable snapshot. Atomic commit = exactly one logical change, with the codebase building and tests passing at every commit. Conventional Commits = a machine-readable intent tag type(scope): description with types feat / fix / refactor / docs / test / chore / ci / perf and BREAKING CHANGE / !. PR = a request to merge a branch with a review UI. Draft PR = opened early, not ready to merge.

Why atomic + conventional
PropertyBenefit
Atomic historygit bisect can isolate the exact bug-introducing commit
Atomic historygit revert undoes one change cleanly, no collateral
Conventional typesAuto changelog + semantic version bumps
Conventional typesCI can skip deploys on docs-only commits
Atomic + conventionalReviewers see why, not just what
In plain words

git bisect binary-searches through history to find the commit that introduced a bug — about 10 steps for ~675 commits — but only works if commits are atomic. git revert makes a new commit that cleanly reverses an old one, if that old commit was atomic.

Key insight — the squash-on-merge reconciliation

You want both: frequent small saves (don’t lose work, checkpoint progress) and a clean atomic published history. Squash-on-merge gives both — commit freely with WIP messages on the branch, then squash into one atomic Conventional-Commit message when merging to development / main. The local branch keeps full incremental history; the shared branch gets one clean entry. High conventionalcommits.org

Small-PR evidence — two independent primary sources
SmartBear / Cisco study

2,500 reviews, 3.2M LOC, 10 months: defect detection peaks at 200–400 LOC/review; drops above ~400. Keep pace <300–500 LOC/hr and sessions <60–90 min.

Google eng-practices

~100 lines reasonable, ~1000 too large; the explicit rule is “err on the side of too small.”

Honesty note

The 200–400 LOC finding (Cisco, 2009) has not been independently replicated at scale on modern polyglot codebases — treat it as directionally correct, not precise. High that small PRs review better; Medium on the exact threshold.

Draft PRs: open early (day one of a branch) to start CI and discussion before deep implementation; convert to “ready” when green + complete. Google recommends starting the next change immediately after sending the first for review — treat review wait as non-blocking. That is stacking. High GitHub docs + Google eng-practices

Stacked PRs / stacked diffs — answers pain (c) for big features

Split a big feature into a chain of small dependent PRs, each based on the previous, each independently reviewable and mergeable.

Stacked PR chain
main
 └ feature/step-1   (PR #1: data model, 150 LOC)
    └ feature/step-2   (PR #2: API, 200 LOC)
       └ feature/step-3   (PR #3: UI, 180 LOC)

Open PR #1, immediately start PR #2 on top — review is not a blocking gate. History: Phabricator (Meta, open-sourced 2011) → Graphite (SaaS, auto-rebases the stack) → GitHub native stacked PRs (private preview, April 2026, gh stack). Caveats: needs <200 LOC discipline; without tooling, manual rebasing of children is error-prone; needs TBD maturity. High Graphite docs; Meta via Pragmatic Engineer; GitHub via The Register

05 Automation: auto-commit & auto-PR the balanced view

This directly answers pains (a) and (b) — objectively, including the risks, because you asked for these to happen automatically.

Plain definitions. Auto-commit = a script / agent runs git commit on a trigger without a human deciding. Auto-PR = a script / agent opens a PR via API automatically. Pre-commit hook = a script that runs on git commit and can abort it. CI gate = checks that must pass before merge, enforced by the platform. Squash-on-merge = collapse a PR’s commits into one on merge.

Key insight — the four guardrails that make automation safe

1. Branch isolation — auto-commits go to a private feature branch (e.g. agent/<ticket>-<desc>), never main / development.

2. Squash-on-merge enforced at repo level — WIP becomes one clean atomic commit on merge.

3. CI gate on the PR (branch protection) — broken states can’t propagate even if merged by accident.

4. Draft PR by default — triggers CI, stays out of reviewers’ queue; a human converts it to “ready”.

Trade-off — the pre-commit hook caveat

Hooks >3–5s get bypassed with --no-verify; many run against the working dir, not the staged index. Practitioner rule: fast checks (lint the diff, <3s) in pre-commit; slow checks (full suite, build) in CI on the PR. High for the risks; Medium for the safe-automation pattern — assembled from practitioner sources, no single canonical spec.

Real-world caution

GitHub’s 2026 merge-queue incident reverted commits across 658 repos — automation at scale is not free. Medium The Register reporting.

Plan execution engine — three layers

writing-plans skill

Every plan task embeds a git commit step (per-task commits are designed in); then offers Subagent-Driven (recommended) or Inline execution.

subagent-driven-development skill

One implementer per task, sequential (parallel implementers explicitly forbidden as a Red Flag); each implementer implements → tests → commits → self-reviews; then spec + quality reviewers per task; a final reviewer after all tasks. “Parallel” here = parallel reviewers, not implementers. Continuous execution, no check-in between tasks.

parallel-plan-execution.js (true parallel engine)

A Workflow, not an inline skill. wave-planner topologically sorts tasks by dependsOn + fileScope overlap into waves (overlapping scopes can’t share a wave). Two isolation modes: worktree (each task its own worktree + branch; an integration agent merges task branches into baseBranch with --no-ff after each wave; worktrees removed) and scope-fence (single wave, agents work uncommitted in the main tree; a fence agent verifies only declared scopes touched; changes stay uncommitted by design).

Phases: Waves → Integrate → Boundary → Final Review. Boundary validation (fullValidationCmd) runs once after all waves (line 233). Final review by code-reviewer. No automatic push; no gh pr create.

route-planner.mjs

Picks the lane workflow / light / inline from task count, wave count, context saturation, etc. No git ops itself.

using-git-worktrees skill

Worktrees at .worktrees/<branch> off a baseBranch (assumed a feature branch, not main directly).

finishing-a-development-branch skill

A 4-option menu: (1) merge locally, (2) push + gh pr create, (3) keep, (4) discard. Option 2 is the only path to a PR and requires a human keystroke.
