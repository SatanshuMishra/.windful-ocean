00 Bottom line up front the design in four frames

0.1 · The target, one sentence

Move the global ~/.claude parallel-development workflow to an MSP-driven model — the unit of work is a Minimum Shippable Product (one coherent, independently-shippable change); one PR per MSP; parallel-safe at every level (git worktrees + solo); adaptive per-project branching; clean per-MSP published history.

0.2 · The core architectural insight — a strict two-level model

Level 1 — INTERNAL (offline, engine, mechanism unchanged): ephemeral task branches merge --no-ff onto the MSP feature branch. Per-task attribution is preserved here.

Level 2 — PUBLISHED (network, NEW publisher): the MSP feature branch lands on the declared integration branch as ONE squash commit, via a serialized merge-queue + Conventional-Commits title lint.

The engine never touches the published integration branch. All network/destructive mutations live in one new, auditable publisher component. Today’s engine already merges task branches --no-ff onto baseBranch (which IS the MSP feature branch) — so Level 1 already exists; the work is Level 2 + a mid-run validation fix.

0.3 · Two locked decisions Locked
  • L5 — per-MSP squash at the published boundary.
  • L4 — branch contract “declare-or-pass-or-ASK, NEVER default” for BOTH source and base.
0.4 · The single genuine fork Open

What is one engine RUN relative to an MSP? Recommendation: Option C — 1:1 (run = one MSP) as the default, but build the publisher/queue MSP-keyed so 1:N is a later annotation change, not a re-architecture.

01 The MSP model what we are building, and why two levels

MSP (plain words): the smallest change that is independently shippable — it builds, passes CI, and delivers value on its own. It is the unit that becomes one PR.

Task vs MSP

A task (a single annotated work item in a plan) is SMALLER than an MSP. An MSP is one-or-more tasks that together ship. The agreed shape, from the locked direction decision (2026-06-26-msp-driven-direction): one PR per MSP; parallel-safe at every level incl. worktrees + solo; adaptive per-project branching; per-MSP squash history.

1.3 · Why the offline / network split is load-bearing

Keeping the engine deterministic + offline means it stays testable (no network, no force-push surface). Confining every allow-listed mutation to ONE publisher component preserves an audit trail — the same “human operates the live surface, agent operates static artifacts” posture as the no-direct-DB rule. The engine produces a validated branch; the publisher (and ultimately the human) lands it.

03 The two locked decisions decided this session Locked

L5 Per-MSP squash Locked

WHAT: the published integration branch gets ONE squashed commit per MSP; atomic commits remain on the MSP feature branch.

WHY: clean integration history, one-command whole-MSP revert, reliable bisect at MSP granularity, release-tool fit (release-please recommends squash). github.com/googleapis/release-please · GitHub: about merge methods

COST (stated honestly): loses sub-PR git bisect granularity in published history — bounded; feature branch retained for archaeology; mitigated by keeping MSPs small.

WHERE IT APPLIES: ONLY at the MSP→integration published boundary (gh pr merge --squash --subject "<CC>"). The engine’s intra-run --no-ff merges are UNCHANGED. Two different boundaries; no conflict.

REQUIRES: if Conventional-Commits / semantic-release is used, the squash message (PR title/subject) MUST be CC-formatted → a new PreToolUse title-lint hook. conventionalcommits.org · semantic-release #1581

L4 Branch contract — “declare-or-pass-or-ASK, NEVER default” Locked, with the user’s modification

THE RULE: for BOTH the source/head AND the base/target of any PR/merge, resolve in this order: (1) EXPLICIT pass (caller states it) → (2) DECLARED machine-readable config → (3) STOP AND ASK.

NEVER derive the base from the platform default branch. NEVER assume the source. Defaulting a PR onto master / the platform default is the CRITICAL failure this forbids.

WHY robust (industry-grounded): heuristic name detection is NOT industry standard — GitHub REST makes base REQUIRED; gh pr create --base is explicit; semantic-release’s hardcoded name list proved fragile at master→main. GitHub REST: pulls · gh pr create · semantic-release #1581

HONESTY GUARD (ties to code-is-truth): a declared integrationBase is validated against LIVE git (the named branch must exist); if the declaration is stale/missing, FAIL CLOSED to ASK. The config is authority only AFTER git confirms the branch exists — “verify recalled specifics against code.”

4.3 · The core design principle

The offline-engine / network-publisher split keeps the engine deterministic + testable and confines all allow-listed mutations to one auditable place.

05 The one genuine fork run-to-MSP granularity Open

Levers L2b (queue) and L6 (semantic-conflict surface n(n-1)/2) only have meaning ACROSS multiple MSPs. So the architecture must support several MSPs on a shared integration branch — but whether they come from ONE run or MANY runs is the fork.

OptionFit w/ current engineComplexityRiskTrade-off
A — Run = MSP (1:1)Very high (baseBranch already IS the MSP branch)LowLowqueue often length-1 for solo; multi-MSP throughput comes from multiple worktrees
B — Run = many MSPs (1:N)Low (rewrites merge step + wave loop)HighHighmost faithful to “small PRs from one plan”; large engine rewrite; L1 becomes heavy partitioning
C — Hybrid: 1:1 now, MSP-keyed publisher from day oneHighLow-MedLowslight extra upfront (key everything by mspId); avoids a future engine rewrite
Recommendation Recommended

Option C. Default behavior == A (simplest, least churn, immediately shippable), but the publisher + L6 graph are MSP-keyed so 1:N later is an annotation change, not a re-architecture. Reject B now (biggest, riskiest change before foundations prove out); reject pure A (would tempt a 1:1-only queue needing rework when worktree concurrency arrives).

06 How the six levers compose new vs reused + integration point Recommended

L1MSP unitNew annotation field

mspId grouping + MSP-level dependsOn in parallel-plan-annotation (schema today at SKILL.md:14-23 has no grouping). Under Option C default (1:1), the whole run is one MSP = trivial single tag. Prerequisite for L2b / L3 / L6.

L2aInter-wave validationNew engine block

Insert a validation call INSIDE the wave loop after each merge (~:215), reusing the boundary-validation agent pattern (:233). Fixes “a broken task poisons the shared branch mid-run.” Independent of the MSP model — valuable TODAY.

L2bMerge-queueNew component

In merge-queue.mjs; each MSP is updated to the integration tip and re-validated before its squash lands — exactly GitHub merge-queue semantics (“grouped into a merge_group with the latest base_branch as well as changes from PRs ahead of it”). GitHub: managing a merge queue. Serialization turns the quadratic pairwise-conflict problem into n sequential validations.

L3Draft-PR + allow-listNew skill + settings

Four guardrails compose as independent layers — private branch (push only the MSP source), squash (--squash), CI-gate (gh pr checks before merge; branch protection is a human post-step), draft-default (--draft + gh pr ready as the human consent gate). gh pr merge --squash --subject sets the squash subject explicitly. gh pr merge

L4ContractThreaded end-to-end

source = MSP branch (explicit pass from the run); base = msp.config integrationBase (declared) OR --integration-base flag (explicit, higher precedence) OR STOP & ASK. Never platform default.

L5Per-MSP squashPublish-step property

Realized at gh pr merge --squash --subject "<CC>" + the cc-pr-title-lint hook. Not a separate phase.

L6MSP dependency-graph gateReuse

Reuses wave-planner’s topo-sort (planWaves :30-58) applied to a new node set (MSPs, not tasks) → produces queue ORDER. L2b’s validate-on-validated-base is the safety net for UNDECLARED edges (semantic conflicts). L6 reduces probability; L2b catches the miss. Defense in depth.

6.1 · Pressure-test answers — resolving the dangerous-looking interactions

Squash vs intra-run --no-ff

Squash applies ONLY at the published boundary; engine --no-ff merges are mechanically unchanged (they target the MSP branch, already true). Engine delta for squash = NOTHING; the real engine delta is L2a, which is orthogonal.

Merge-queue vs validate-once

Two mechanisms at two levels. L2a validates BETWEEN WAVES within a run (engine); L2b validates BETWEEN MSPs against the integration tip (publisher queue). Throughput cost of serialization mitigated by: re-validate only when the integration tip moved; parallel-validate L6-independent MSPs; length-1 fast path for solo single-MSP work.

Declared config vs code-is-truth

A machine-read JSON consumed deterministically is FUNCTIONAL INPUT (like runArtifacts / settings.json), not prose that drifts. Honesty guard: validate the declared branch against live git; fail closed to ASK.

L6 vs existing dependsOn

REUSE of the same format/algorithm on a new node set. Task dependsOn orders tasks within an MSP + guards textual (fileScope) overlap; MSP dependsOn orders MSPs in the queue. Rule: a task dependency crossing MSPs implies an MSP dependency. Semantic conflicts (disjoint files, interacting behavior) are caught by L2b’s validation against the cumulative integration tip before the squash lands.

Draft-PR guardrails + allow-list

Draft-default gives CI time + a human un-draft gate; CI-gate blocks premature merge; squash collapses to one commit. Independent, non-conflicting layers.

10 Constraints & guardrails recap the non-negotiables

  • ~/.claude is NOT a git repo: every change is a static config artifact (rules/skills/engine JS/hooks/settings), applied via files; protect-claude-config.sh returns “ask” on rules/skills/settings writes — human approves.
  • This workflow GOVERNS how Claude Code behaves in the user’s OTHER projects; it is authored here, run there.
  • No auto-DB / cloud-admin; the human operates live surfaces (branch protection, merges to protected branches).
  • The four auto-PR guardrails are mandatory (private branch, squash-on-merge, CI gate, draft-default) — auto-PR is net-negative without all four.
  • No code comments, no emojis, no AI attribution in any authored artifact.
