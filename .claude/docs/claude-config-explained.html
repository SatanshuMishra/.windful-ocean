01 Overview what is all this?

This configuration turns Claude Code from a single chat assistant into a small, governed software organization. A single orchestrator (the main Claude in the session) almost never writes code itself — it routes work to a fleet of specialized subagents, each in its own fresh context. Surrounding that fleet are four control layers that all fire automatically: rules that constrain every task, skills invoked when a matching task appears, hooks that block or react at fixed lifecycle moments, and plugins / MCP that add capabilities and external-tool connectors. Three persistence systems — auto-memory, the continuity ledger, and graphify — carry context across sessions. The net effect: a feature request flows through a fixed pipeline of design → plan → delegated implementation → review → verification → commit, with safety rails enforced deterministically rather than relying on the model to remember them.

02 Skills the playbooks

A skill is a saved procedure Claude loads and follows step-by-step when a matching task appears. Skills are triggered three ways: the owner typing /<name>, the model auto-matching a task to a skill's description (the using-superpowers rule forces a skill-check before any response), or a hook auto-loading one. Skills compose into chains — that chaining is the heart of Section 7.

03 Agents the worker fleet

A subagent does the actual work in fresh, isolated context and reports back a distilled result, so noisy logs and file dumps never pollute the main conversation. Under the delegation discipline rule the main thread dispatches a subagent even for a one-line typo fix — a deliberate ~5–10k-token round-trip — because the orchestrator's job is judgment, not typing.

Model strategy. The deepest-reasoning roles use opus (architecture, debugging, both reviewers, performance, data, security — judgment-heavy and read-only). General builders use sonnet (analysis, research, tests, docs, strategy). Rote edits use haiku (cheap, fast). The implementer uses inherit — it runs at whatever model the session is on (here, Opus 4.8). This matches the owner's performance.md model-selection rule.

Things to notice The reviewer / analyst / architect / researcher agents are all read-only — they never edit. Only implementer, mechanical-editor, debugger, data-engineer, devops-engineer, and technical-writer can write. Review and authoring are strictly separated.

04 Rules the always-on constitution

A rule constrains behavior on every task, no invocation needed. CLAUDE.md (the global-invariants hub, loaded every session) sits at the top; it points to rules/common/* (apply everywhere) and rules/typescript/* (loaded for TS/JS work). Precedence: user rules outrank skills outrank the default system prompt — so these files are the highest authority in the session.

Six hard invariants — no exceptions
  1. NEVER write code comments (derive understanding from raw code; stale comments mislead).
  2. NEVER use emojis in code / commits / docs / UI unless asked.
  3. NEVER add AI co-author attribution.
  4. NEVER connect directly to live databases or cloud-admin surfaces (author static SQL; a human runs it).
  5. Destructive shell / git operations require explicit confirmation.
  6. Tool routing: graphify to orient → Serena for precise symbol nav → grep / Read for local / quick.

05 Hooks the automated reflexes

A hook is a script the system runs automatically at a fixed lifecycle event. Hooks are deterministic — they fire whether or not the model "remembers" to. Three behaviors matter: HARD-BLOCK denies the action, PROMPT asks for confirmation, and WARN/LOG only nudges. A few are passive bookkeeping, background, or sound.

Things to notice Only THREE hooks truly hard-deny (secret-scanner, pre-commit-scoped-verify, block-deep-research). The destructive-bash and protect-config hooks PROMPT rather than block — they add a confirmation step, not a wall. And block-env-edits.sh uses exit code 1 where Claude Code's documented hard-block code is exit 2, so it may surface as a non-blocking error rather than a true deny (marked block* above). A one-line change to exit 2 would make it a real block — adjust via a hooks-editing skill if desired.

Cross-cutting systems

Auto-memory

Durable per-project facts (decisions, constraints) that aren't in the code; stored as one-file-per-fact with an index.

Continuity ledger

Per-project thread files (PROJECT.md, threads, decisions, sessions) with a 5-state lifecycle and a Definition-of-Done gate; written by session-handoff, read by resume-project.

graphify

A locally-built knowledge graph of the codebase, provisioned at session start and refreshed on Stop; queried token-free to orient before drilling in.

Tool routing

Map → GPS → Street view: graphify to orient, Serena for precise symbol facts / edits, native grep / Read for local known-location work.

Things to notice The integration set is deliberately read-or-author-only at the boundary — github (scoped token), serena / context7 (local / docs), playwright (local browser). The absence of any DB / cloud MCP is intentional and enforced by both a rule and the deny-list.

Cross-cutting — active at every step
Tool routing governs every read: graphify (orient) → Serena (precise) → grep / Read (local).
no-direct-db-access forbids touching any live DB / cloud at every step; the deny-list backs it.
Permissions gate every shell action: 22 pre-approved, 29 hard-denied.
verification-before-completion is not a single step — it guards EVERY "done" / commit / PR claim throughout.
The five human gates The pipeline is mostly autonomous between gates. The human is asked at five points: approve the design, approve the written spec, (informationally) the wave layout, the writing-plans execution-mode choice, and the final merge / PR / keep / discard decision. Everything else — implementation, review, debugging, verification — is delegated to subagents and enforced by hooks.
