This report is the visual rendering of the authoritative design document. Everything marked as a decision-to-confirm, an open question, or a tradeoff is presented AS such below, not as settled fact. v2 incorporates seven follow-up concerns from the user and three new research probes (F: MCP/plugin mechanics; G: durable git-metadata storage; H: generic work-unit modeling).

01Why this exists
BLUF plus the three original problems
Bottom line up front

The current session-continuity "Continuity Ledger" is being rebuilt from the ground up into a componentized, published plugin installable into ANY Claude Code config, not something hard-wired to the author's personal setup. It fixes three original problems: it is single-user only, its robustness lives in fragile prose, and its "tree-branch" architecture never actually linked anything.

The v2 design converges on: one recursive Thread data model; a bundled MCP server as the sole, schema-enforcing writer; a tool-owned orphan branch (a decision to ratify) as the durable store; and a drift pipeline reframed from "police bad git states" to "reconcile the ledger with reality."

1Single-user only

The ledger is committed for git projects but has ZERO merge / conflict / drift logic. A teammate's pull can clobber work; resume then trusts a stale ledger. ("Are we rebuilding Git?")

2Robustness via prose

All formats and rules (thread YAML, the state machine, caps, write-once, filename schemes) live in SKILL PROSE parsed by awk hooks. Rename a field and a silent empty match breaks the scan. About 8 unenforced drift points. The redesign wants a schema / tooling GUARANTEE.

3Architecture broken

The old "tree-branch" language never linked threads; open and closed threads shared one directory; there was no real Project ↔ thread ↔ decision relationship. The redesign wants a real, scalable hierarchy.

02Goals and non-goals
Concern 1 — the durability reframe (this changes everything downstream)

The core promise is often mis-stated as "the ledger survives branch deletion." The precise, corrected goal is narrower and it reshapes the rest of the design.

GOAL — survive the NORMAL lifecycle of a branch

A feature branch is routinely PRUNED after its work completes (post-merge cleanup, periodic housekeeping). The ledger's recorded context must still be present afterward, so that related future work — a new session, or a teammate who no longer has the original branch — can recover the earlier decisions, session history, and pointers.

NON-GOAL — policing bad practices

Defending the ledger against deleting a branch MID-FEATURE, force-pushing over history, or deliberately erasing the record is explicitly OUT OF SCOPE. That is an adversarial / bad-practice scenario, not the design target. No alarm machinery is built for it, and it does not drive storage or lifecycle decisions.

Why the reframe matters architecturally (validated by Probe G)

Once "survive deliberate deletion" is off the table, the durability bar drops to "outlive normal branch cleanup." No storage backend defends against a direct git push --delete of its own ref, so tamper-resistance was never achievable anyway. The decision therefore reduces to three practical axes — zero-config propagation, host portability, and UI cleanliness — not "how hard is it to destroy."

Consequence for the drift pipeline (Section 6)

Its FRAMING shifts from "detect and warn about dangerous git states" to "reconcile the ledger with reality and recover context gracefully after normal events (merge, squash, prune, rebase)." Same signals, calmer purpose.

03The unified recursive data model
Concerns 5 and 6 — collapse Tree / Thread / Leaf into ONE recursive Thread (Probe H)

The flaw in v1

v1 declared a fixed three-level hierarchy Tree(project) → Thread(line of work) → Leaf(shippable unit), and said a Leaf "maps to an MSP when the mitosis skill is present." Two defects:

  • Over-coupling (concern 5). "MSP" / "mitosis" is ONE workflow tool in the author's personal config. The plugin must never behave specially because a particular tool is installed — it must work identically for any config.
  • Over-ceremony for small work (concern 6). Tree → Thread → Leaf is right for a large multi-part effort but absurd for a single fix over two sessions.

The fix — one recursive entity Probe H

Collapse Tree / Thread / Leaf into ONE recursive entity. The system already calls a line of work a Thread; keep that word and make it recursive. There is no separate "Leaf" or "Tree" schema. Convergent evidence: Linear, GitHub sub-issues, git-bug, Fossil, and MADR all model a unit of work with ONE entity shape reused at every depth plus a nullable self-referential parent. Jira — the one system with a fixed type-per-level hierarchy — is the documented cautionary tale (teams "outgrow" the fixed levels and must migrate).

AOne shape at every depth

A Thread is a Thread whether it stands alone or contains children.

B"Has children" is a QUERY

Never a stored flag. A Thread is an "epic" iff some other Thread has parent_id pointing at it. This removes the two-data-model problem entirely.

CLazy promotion

Start flat. A standalone fix is ONE Thread with parent_id = null and no children. Promote to a parent only when a second related unit actually appears — never pre-declare hierarchy.

DAdjacency list

A parent_id pointer, not a materialized path or nested set — correct for a shallow, dozens-of-nodes, append-heavy scale.

Decoupling from any workflow tool concern 5

The core auto-detects and models work from UNIVERSAL signals only. Anything workflow-specific enriches the model through ONE generic, optional extension point — the external_refs bag — that the core never inspects.

The generic extension point

Tool-specific concepts attach only via a generic external_refs bag on the Thread: a list of { system, id, url }. The core stores it opaquely and never reads it. mitosis becomes ONE possible implementer of a capability-detected interface — indistinguishable in the core's eyes from a Jira integration, a Linear integration, or nothing at all. This mirrors LSP capability negotiation, the dependency-inversion principle, and the config's own /verify-<project> convention ("call it if it exists, else fall back"). If a project declares a well-known "unit provider," the plugin may call it to suggest a label / status, a parent_id, or populate external_refs. Absent → silent no-op.

04Storage architecture
Concern 3 — where the pointers and prose actually live

The ledger stores POINTERS and PROSE, never code: Thread / BranchBinding JSON, MADR decisions, append-only session logs, a derived index, and PROJECT.md. For a multi-user git project this state must (a) be shared across clones, (b) outlive normal branch pruning, (c) require no manual sync step per teammate, and (d) work on whatever git host the team uses.

The core tension Probe G

Property neededReal branch refs/heads/*Custom ref refs/ledger/*
Auto-syncs with zero config (default fetch refspec transfers it)YesNo
Stays OUT of the branch list / GitHub UINoYes
Guaranteed pushable / fetchable on EVERY git hostYesHost-dependent (git-bug friction precedent)
Idiomatic among git-native tools (6-tool survey)Zero tools use a plain orphan branchThe community idiom

These two properties are mutually exclusive for the same ref. No ref type gives both for free. Probe A independently flagged that a plugin creating a branch in a user's repo is "unusual — must be visible, opt-in, documented, and cleaned up," not a silent side effect.

Two facts that keep the orphan branch viable

1Host portability (Pillar 1: robustness)

A branch is guaranteed pushable / fetchable on EVERY git host. A custom ref namespace depends on the host accepting non-standard ref pushes — widely supported (GitHub, GitLab) but NOT universally guaranteed. This is the strongest argument for the branch.

2The MCP server neutralizes the sync burden

Probe G rejected the custom ref only because "someone must remember to run the sync step." Here the sync is run by the MCP server automatically every session (SessionStart reconcile + Stop handoff both fetch / push the ledger ref). No human remembers anything, so the custom ref's one disadvantage does not apply to us.

Decision to ratify — the single biggest open decision for the user

Default: a single, tool-owned ORPHAN BRANCH, treated as owned infrastructure — the exact posture GitButler takes with its gitbutler/workspace branch ("an internal branch the tool owns; don't touch it"). Named distinctively so it sorts away from feature branches (e.g. _ledger or ledger/state; configurable). The MCP server is its sole writer and accesses it via git worktree add. Because it is orphan, pruning / squashing / deleting any feature branch is a NON-EVENT for it.

Documented opt-in alternative: a custom ref namespace refs/ledger/* for teams that prefer a clean branch list AND are on a host known to accept custom refs. The MCP server auto-installs the fetch refspec (guarded against clobbering an existing remote.origin.fetch) and owns push / pull, so it stays zero-touch. The storage backend sits behind the MCP tools, so switching is a config choice, not a data-model change.

This hardens (does not reverse) v1's orphan-branch decision, drops the now-moot "GC-proof / deletion-immune" claims, and names it as infrastructure per Probe A / G etiquette. The one honest cost of the default remains cosmetic: the infra branch appears in git branch -a and the host's branch dropdown — a trade taken deliberately (robustness + zero-config over branch-list tidiness).

Non-git projects concern 7

Git-ref storage is impossible without git. For a non-git project (e.g. ~/.claude itself), the ledger falls back to a LOCAL, per-project store at ${CLAUDE_PLUGIN_DATA}/<project-key>/ledger/, where CLAUDE_PLUGIN_DATA persists across plugin updates and <project-key> derives from the absolute CLAUDE_PROJECT_DIR path (always set). Consequences: single-user and machine-local (acceptable — a non-git project has no shared remote anyway). BranchBinding and the git drift pipeline are simply INACTIVE; Threads carry vcs_ref = null and the model degrades gracefully. The SAME data model, SAME MCP tools, SAME skills — only the storage driver differs, selected by "is CLAUDE_PROJECT_DIR a git work tree?" This formalizes how the current system already treats ~/.claude.

The feature branch carries NO ledger files, so deleting / switching feature branches never touches the ledger, and ledger data is never accidentally committed into feature history. Placement leans OUTSIDE the repo (under ${CLAUDE_PLUGIN_DATA}, no .gitignore needed); if in-repo it must be excluded via .git/info/exclude — to be pinned at spec time. The bridge from "I am on branch X" to "this is Thread T" is the Thread-Id: commit trailer plus the derived branch → binding → thread index, never a working-tree file.

Concurrency

The MCP server serializes writes per machine. Cross-machine it does fetch → auto-merge → push with CAS (compare-and-swap) retry on a non-fast-forward race. All shared files are per-session / per-thread / per-binding and therefore DISJOINT, so concurrent pushes auto-merge with no content conflicts. A merge=union driver on append-only text files is a belt-and-suspenders option.

05Enforcement planes
Concern 4 — what MCP is, and whether we are building one

The robustness guarantee (original problem 2) comes from moving the ledger's rules OUT of skill prose and INTO tooling. Two planes do this: an MCP server (data plane) and hooks (control / lifecycle plane).

Are we creating an MCP? — Yes

MCP (Model Context Protocol) is an open standard for exposing tools to an AI agent. In Claude Code an MCP server is NOT a website — it is a LOCAL SUBPROCESS the plugin ships. We author and bundle one small stdio program (Node, Python, or a compiled binary) that Claude Code launches as a child process and talks to over stdin / stdout using JSON-RPC, declared in a .mcp.json at the plugin root. Its tools reach the model as native tools named mcp__ledger__<tool>, with arguments and results as schema-validated JSON. One instance starts when the session begins and lives the whole session; the first call prompts for approval, then auto-approves.

It is the SOLE reader / writer of the ledger. Because every mutation goes through a typed, schema-validated, FSM-aware tool, malformed ledger state becomes structurally impossible — the model never hand-writes files at all. It calls transition_thread(id, "done") and the server enforces the DoD gate, caps, write-once, and atomic write.

Honest tradeoff (Pillar 1 vs portability)

The MCP server is a RUNTIME DEPENDENCY — the machine needs the chosen runtime (Node or Python) or we ship a compiled binary per platform. A pure-hooks-plus-CLI design would avoid the daemon but reintroduce stringly-typed, permission-prompted, parse-prone interactions and lose the schema guarantee. We choose the MCP server because the guarantee is the whole point of the redesign; the runtime cost is documented and mitigated by shipping a binary or requiring Node (already common).

The Thread-Id: commit trailer + auto-installed hook Probe G — Gerrit precedent

To bridge a branch to its Thread durably, the first commit of every feature branch carries a Thread-Id: <ulid> trailer. This is directly modeled on Gerrit's Change-Id: a stable identity in the commit-message footer that survives amend / rebase / cherry-pick, discoverable via git log --grep="Thread-Id:", and thus recoverable even after the branch is deleted. A commit-msg hook inserts the trailer if absent, does nothing if present (idempotent → amend-safe), and is opt-out via git config.

The distribution lesson — and its clobber caveat

The critical real-world lesson from Gerrit is DISTRIBUTION: Gerrit still makes users manually curl + chmod the hook into every clone, and "missing Change-Id" rejections are the common result. A distributable plugin must AUTO-INSTALL the hook via core.hooksPath (Git 2.9+) pointing at a plugin-managed hooks dir, NOT by hand-copying into .git/hooks/. Caveat: if the user already sets core.hooksPath (Husky, pre-commit), the plugin must CHAIN / APPEND rather than clobber. The trailer is a convenience for re-attach, not a correctness gate — its absence degrades to slug / manual re-attach (Section 6.4), it does not corrupt state.
