02

Why this exists — the problem

A Claude session (one continuous conversation) has a finite context window (the amount of text the model can “see” at once — yours is a 1M-tokentoken: the unit of text a model processes — roughly a word-piece. Window size is measured in tokens; yours holds about one million. Opus window). Two things destroy continuity: (1) the window fills up and Claude Code compactscompaction: when the window fills, Claude Code auto-summarizes the old turns and throws the detail away to make room. Detail is lost exactly when work is deepest. it — auto-summarizes the old turns and throws the detail away; (2) you end the session and start a fresh one tomorrow with a blank slate. In both cases the reasoning, decisions, and “where I left off” evaporate unless they were written somewhere durable. The ledger is that durable somewhere.

Three hard facts the design assumes
1

Memory is not persistent

A new session knows nothing the files don’t say. So the files must be enough to resume from files alone.

2

Context is finite & gets compacted

Detail is lost exactly when work is deepest. So the system must checkpoint before the cliff, automatically.

3

Resuming the WRONG work is the worst failure

Silently auto-picking the most-recently-touched task and charging ahead is the cardinal sin. So resume always presents and stops.

03

The mental model: two layers

The single most important architectural idea is that the system is split in two on purpose: a deterministic layer and a judgment layer.

  • The deterministic layer = hooks. Small shell scripts the harness runs automatically at fixed moments. They never decide what your work means — they only fire on schedule and either save a tiny breadcrumbbreadcrumb / sentinel file: a tiny file a hook writes to disk to mark state (e.g. that a compaction is about to happen), to be read back later. to disk or injectinject (additionalContext): a hook adding text directly into the chat context as a reminder, without you typing it. a reminder into the chat. They are the reflexes.
  • The judgment layer = skills. session-handoff and resume-project. These read your actual work, decide what matters, and write the real ledger files. They are the thinking.

The seam between them: a hook can prompt the skill at the right moment with the right facts, but the model (and you) still decide what to do.

The seam

Every hook is a trigger, never an author. The actual ledger content is always written by a skill. This is why a prompt-injected or confused hook can’t corrupt your record.

Anatomy of the ledger — what gets stored

4a · Where it lives
Project type Ledger location

Non-git ~/.claude/projects/<project-slug>/ledger/ — kept under your home config

In a git repo the ledger is committed (so it clones/branches with the code); the hand-off commit message is exactly chore: ledger handoff <thread-slug>. Non-git projects are written but never committed.

4c · The five thread states

A thread (one line of work) is always in exactly ONE state.

State Meaning How it’s entered Required field
active Being worked in this session, right now New thread, or resumed completion_criteria (at creation)
paused Parked, resumable Automatic at every handoff (active → paused) —
blocked Waiting on a dependency Explicit blocked_by
done Finished Explicit, only if DoD gate passes all completion_criteria checked + closure line
abandoned Dropped Explicit abandoned_reason
The “active is an anomaly” rule

active means this session only. So if a later session ever finds an active thread, something crashed — that’s a “zombie,” and the staleness scan flags it (Section 7).

The Definition-of-Done (DoD) gate

Claude will refuse to mark a thread doneDoD gate: the Definition-of-Done check — a thread can only be marked done if its completion criteria are non-empty, all checked, with a one-sentence closure line. unless its completion_criteria list is non-empty, every item is checked, and there’s a one-sentence closure. Criteria are set when the thread is created, never invented at the end. This stops “done” from meaning “I got bored.”

4d · The “spine” — why resume stays cheap

Each thread file carries a fixed set of running-summary fields — the spine: Status, Active Goal, Next Step, Open Risks, Key Decisions (links only), Out of Scope, Pointers, Recent Sessions. At each handoff Claude merges the old spine with the new session log into a refreshed spine. So whether a thread spans 2 sessions or 20, the thread file stays one short, current page — not an ever-growing transcript. Decisions are never compressed into the spine; the spine only links to the decision sidecar files by filename.

Two cross-cutting disciplines

Pointers, not payloads. Ledger files carry paths and links, never copied-in file contents.

Caps + demotion, never deletion. When a file exceeds its line cap, the overflow is demoted into the current session log with a pointer left in its place. Nothing is deleted outright.

05

What happens when you HAND OFF — the write side

You trigger it by saying “session handoff” / “wrap up session” / “hand off”, or by confirming the auto-nudge when context fills up. Then Claude runs an ordered 11-step protocol.

# Step Reads / writes
1 Wind down — collect results from any running subagents/background tasks, or stop them cleanly. Never abandon a write mid-flight; anything still running is recorded with shell IDs + kill commands. reads session state
2 Locate or initialize the ledger — resolve the dir (git vs non-git); if absent, create PROJECT.md + empty threads/decisions/sessions. Idempotent: never overwrites an existing file. writes (maybe) PROJECT.md + dirs
3 Append the session log sessions/YYYY-MM-DD-NN-<slug>.md — the narrative, including what FAILED and why.
What’s inside a session log
  • Where it started — the entry state for this session
  • What shipped — the concrete delta
  • Tried and failed — dead ends and why they failed
  • Verification — the commands run to prove it
  • Running state — shell IDs + kill commands
  • Deferred + open — what was punted
  • Pick up here — the precise next stopping point
 writes new session log
4 Promote stragglers — any decision locked this session that has no decision record gets one now (safety net; ideally captured at decision-time).
Decision-record template + supersession rule

Template fields: Status / Date / Thread / Context / Decision / Consequences.

Supersession rule: a new decision = a new file. The old record’s only allowed edit is its Status line, changed to superseded-by: <filename>. The original is never rewritten.

 writes 0+ decision records
5 Refresh the thread spine + apply the state transition — merge old spine + new log into the refreshed spine fields; default transition active → paused. reads old thread + new log; writes thread
6 DoD gate (only if closing done) — verify completion_criteria non-empty + all checked + closure line; else REFUSE done, leave paused/blocked, say why. decision gate
7 WIP check — if more than one unrelated non-terminal thread exists, surface it and recommend disposing the stragglers. Never auto-close. reports
8 Touch PROJECT.md — only if project-level facts changed (new decision index line, thread added / status changed, new constraint). writes (maybe) PROJECT.md
9 Enforce caps — PROJECT.md/thread 80 lines, decision 20; demote overflow into the session log with a pointer. Nothing deleted. rewrites to cap
10 Commit (git only) — commit only the ledger files: chore: ledger handoff <slug>. Your wrap-up confirmation is the consent. Non-git: write only. git commit
11 Print the chat summary — the human-facing courtesy block (below). prints
The chat summary block (the courtesy copy)
SESSION HAND-OFF — <thread-slug>
Shipped this session:  <the delta>
Thread + state:        <slug> — active -> paused
Handed off (next):     <the single next action>
Resume command:        /resume-project <slug>
Open / blockers:       <if any, else "none">
Files written:         <ledger paths>
Then: /clear
Files vs chat — what’s the difference?

The chat block is a compressed snapshot for you; it is NOT the record. The canonical record is the four files. The chat block even adds two things that are deliberately not written to any file: the /resume-project <slug> command and the Then: /clear instruction — operational affordances, not part of the saved state.

06

What happens when you RESUME — the read side

You trigger it by saying “continue” / “resume” / “pick up where we left off” / /resume-project <slug>.

# Step What it does
1 Locate the ledger git path first, then the global fallback. Missing → tell you, offer to initialize via session-handoff, stop.
2 Read PROJECT.md the only file read unconditionally besides the chosen thread + its latest log.
3 Select the thread deterministically explicit /resume-project <slug> → use it, skip the menu. Otherwise → present a menu of every resumable thread (active/paused/blocked), one line each, and STOP for you to choose. Never auto-select by recency.
4 Load exactly two more files the chosen threads/<slug>.md + its single most-recent session log. Nothing else. Decisions load on demand later, by the filename in PROJECT.md’s index.
5 Verify against code + git run git log --oneline -5 (or file mtimes) vs the thread’s updated date; flag work done after the last wrap-up. Spot-check the thread’s Pointers exist on disk. On divergence: code wins — note it in the brief, fix the ledger.
6 Present the Resumption Brief, then STOP show the brief (below) and wait. Do not proceed, do not offer “which path?” prompts.
The token budget — how resume re-acquires context cheaply

The hard read budget is PROJECT.md + one thread file + one session log, ≤ 3k tokens. It deliberately does not bulk-read decisions/ or sessions/. That’s the payoff of the spine + pointers design: a thread that spans 20 sessions still resumes from ~3k tokens because the spine is always one current page and the deep history stays on disk, loaded only if needed.

The Resumption Brief block
RESUMPTION BRIEF — <thread-slug>            state: <current> -> active (awaiting confirm)
Priority:     <high | medium | low>
Done so far:  <2-3 sentences: what shipped, current state>
Left off at:  <the precise stopping point>
Next step(s): <the single first action, then any follow-ons>
Open / risks: <blockers, open questions>
— Awaiting your instruction. I will not proceed until you direct.
Present, then STOP

The single most important resume rule. The transition paused/blocked → active happens only after the brief is shown and you direct it, never silently. The brief header literally annotates it -> active (awaiting confirm).

Hints, not truth

The ledger is hints; the code is truth. Resume verifies before trusting, and on any mismatch the code wins and the ledger gets corrected.

How context is re-acquired (the answer)

PROJECT.md gives the project shape; the thread spine gives the current state + next step; the latest session log gives the fine-grained “pick up here” + what failed; decision records are pulled by filename only if a specific decision matters. That layered load reconstructs just enough to continue, cheaply.

07

The automatic safety net — the five hooks

Most of the time you don’t invoke any of this — five background scripts watch the dangerous moments and prompt the right action.

Hook Fires on (real moment) Writes / Injects Why it exists
context-wrapup-nudge PostToolUse — after every tool call, watching context fill Writes a telemetry sentinel at ≥50%; injects a wind-down + handoff reminder at 70%, stronger at 80% (each once) Warn before the context cliff so you can hand off cleanly while detail still exists
ledger-precompact-checkpoint PreCompact — just before an auto-summarize wipes context Writes .compact-sentinel-<sid>.json (active thread slug + path to the full pre-compaction transcript). Injects nothing — it’d be wiped Preserve “what was I doing + where’s the full record” across a context wipe
ledger-compact-checkpoint SessionStart (only source=compact) — right after a compaction Reads + deletes the sentinel, injects: “context was compacted; write a mid-session checkpoint NOW (stay active, no /clear)” Re-inject the preserved facts into the fresh context and force a durable checkpoint
ledger-staleness-scan SessionStart (startup/resume/clear) Injects a roster of stale/zombie threads needing disposition (never auto-closes) Make crashed active threads and rotting paused/blocked ones visible instead of silently lingering
ledger-resume-roster UserPromptSubmit — when your message signals resume intent Injects the menu of resumable threads + “use resume-project, present + STOP, never auto-select” Guarantee “continue” always yields an accurate menu, never a wrong-thread guess
The compaction pair — the cleverest bit

ledger-precompact-checkpoint and ledger-compact-checkpoint are one operation split across a context wipe. PreCompact can’t talk to the model (the context it would talk into is about to be destroyed), so it writes the essential state to a sentinel file. The post-compaction SessionStart hook then reads that file back and injects the reminder into the freshly rebuilt context. Neither could do the job alone.

Staleness thresholds — the clock only raises the question

active → flagged always (>7 days idle = “zombie”, else “likely crashed”); paused → flagged after 30 days idle; blocked → after 90 days. The clock only raises the question; you decide disposition. Nothing is ever auto-closed.

Things to notice

The ledger-resume-roster hook keys on the words “continue”/“resume” in your prompt — which is exactly why generating this very report tripped it twice. It’s harmless: the hook only injects a menu; it can’t act on its own, and you (or Claude) simply ignore it when the intent was a false positive.

Design principles — one line each
  1. Files are canonical; chat is courtesy.
  2. Hooks trigger; skills author. (Deterministic vs judgment.)
  3. Pointers, not payloads.
  4. Caps + demotion, never deletion.
  5. Decisions are append-only and never compressed.
  6. Resume presents, then STOPS — never auto-selects, never auto-proceeds.
  7. The ledger is hints; the code is truth.
  8. Nothing is auto-closed; the clock only raises the question, the human decides.
