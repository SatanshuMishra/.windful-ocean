00 Bottom line up front the MSP-driven model in one frame

The MSP-driven model in one frame
  • The unit of work is the MSP: the smallest change that (a) passes CI on its own, (b) leaves the shared branch building + tests-green + behaviourally non-broken when merged alone, and (c) stays dark in production if the larger feature is unfinished (via keystone interface or feature flag).
  • One MSP = one PR, auto-opened as a draft; large features become a stack of small MSP PRs merged bottom-up.
  • “Parallel-safe” means every merge into any shared branch — mainline, a shared feature branch, even across your own three worktrees — leaves it green and the app working. Held by three things: MSP task design, tests covering the interaction surface between tasks, and testing each merge against the actual landing state (merge-queue discipline).
  • Branching adapts per project (detect-then-confirm), degrading gracefully to atomic local commits when there is no repo.
  • History stays clean via branch isolation + one squashed commit per MSP.
Headline contrast — what your config is today

Your current config is a feature-branch batch model. It plans tasks as implementation slices (not MSPs), merges them into one feature branch with --no-ff, validates once after all waves (so the shared branch can sit broken mid-run), opens one human-chosen PR for the whole feature, hard-codes the base branch with no topology detection, and leaves squashing to the PR-platform button.

Excellent for one developer finishing one branch; unsafe for continuous, parallel, MSP-sized shipping.

The single sharpest finding High

In the parallel engine, the per-task check runs inside each task’s isolated worktree, never against the shared base branch after merge; fullValidationCmd runs once at the very end. So a non-MSP task can merge and leave the shared branch broken for everyone else (and your other worktrees) until the final gate. parallel-plan-execution.js: per-wave git merge --no-ff, single boundary validation.

01 The MSP unit the foundation the four clarifications build on

An MSP (Minimum Shippable Product / Piece) is the smallest change that is independently mergeable and non-breaking. The test to apply before planning:

“If this merges today, with everything else still in-flight, does the shared branch build, do all tests pass, and does the app still behave correctly for every journey it supports?” The MSP admission test
What makes a change a valid PR unit High

“Makes a minimal change that addresses just one thing… The system continues functioning properly after submission” and “usually just one part of a feature, rather than a whole feature at once” — Google eng-practices.

INVEST: Independent, Negotiable, Valuable, Estimable, Small, Testable — Agile Alliance. Both primary sources.

Finding MSP boundaries = vertical slicing High / Med

A horizontal slice ships “only the DB layer” or “only the API” — passes tests in isolation but ships no value and often can’t be enabled alone. A vertical slice crosses all layers for one narrow behaviour (e.g. “read-only GET /profile” before “edit profile”).

Decompose: foundational layer (schema / types) at the bottom, one vertical slice per behaviour, the keystone (UI entry point / flag flip) last. The invariant: every slice is green on its own.

Keeping merged-but-incomplete code DARK High

So an MSP is non-breaking even mid-feature, pick one:

  • Keystone Interface — build the backend across PRs, add the UI entry point as the final small PR. Zero flag overhead; first choice.
  • Feature Flag — when the change alters an existing flow rather than adding a clean entry point.
  • Dark Launch — run the backend on real traffic, discard the output, for load testing.

Fowler: Keystone Interface + Dark Launching.

02 Clarification 1 — one PR per MSP auto-opened; large features become a stack

Stacked PRs / base-branch chaining: each MSP is a branch; PR2 targets PR1’s branch (not main); reviewers see only that layer’s diff.

Stacked-PR chain — each PR targets the one below it
main
 └ feat/user-schema   (PR1: migration, targets main)
    └ feat/user-api    (PR2: endpoint, targets feat/user-schema)
       └ feat/user-ui  (PR3: UI keystone, targets feat/user-api)

Cascading restack

When the bottom PR merges, the rest auto-rebase so the next targets main. Merge order is mandatory bottom-up. GitHub gh-stack docs.

The squash-merge hazard

Squash rewrites commit hashes, breaking the chain and creating false conflicts; resolved by git rebase --onto replaying only unique commits. The strongest argument for tooling over raw git. gh-stack FAQ; Sleeping Potato.

Draft-PR-per-MSP

Open a draft PR immediately per MSP; CI runs; convert to “ready” when green (the signal is the conversion, not the creation); cycle-time metrics start at “ready”. Naming: feat(auth): [MSP-2] add GET /users. Convert via PATCH .../pulls/{n} {draft:false}. LinearB; GitHub API.

Counter-case — objectivity High
  • CI fatigue (N MSPs = N CI runs) and reviewer-notification noise; stacks deeper than ~4–5 get hard to track (3–4 for teams new to it; Meta runs 5–10 with heavy tooling).

The invariant High

Every merge into any shared branch must leave it (a) building, (b) tests-green, (c) behaviourally non-broken — not just merges into mainline. example-branch is a shared base for the 3 worktrees, so each task-merge is an integration event that must keep it green.

Fowler, Continuous Integration: “nobody has a higher priority task than fixing the build”; self-testing code is a “necessary prerequisite”; “No code sits unintegrated for more than a couple of hours.”

Three disciplines that hold the invariant

1) Design each task as an MSP (not an implementation slice). Use flag / keystone / branch-by-abstraction / expand-contract so an incomplete task merges harmlessly.

TechniqueUse whenComplexity cost
Feature flagtask adds incomplete visible behaviourMedium toggle debt; logic branches
Keystone interfacebackend first, UI entry point lastLow merge-order discipline only
Branch-by-abstractionreplacing an existing implementationHigh interface layer
Expand-contractchanging an interface / schema other tasks useHigh old+new coexist; expand must land first
Dependency rule

If Task B consumes what Task A adds, they can’t safely parallelize unless A lands first OR A uses expand-contract to publish the new interface while keeping the old live.

2) Semantic conflicts are the real enemy — the key insight for 3 parallel tasks High

A textual conflict = git says “I can’t merge these.” A semantic conflict = git merges cleanly, it compiles, maybe even passes existing tests, but the combined behaviour is wrong.

Honesty note Medium

Fowler: “The harder problem are Semantic Conflicts… the version-control system can’t help us.” Empirical: code in manually-resolved merge conflicts is ~26× more likely to contain a bug (Accioly et al.). This figure is for merge-conflicts-needing-manual-intervention broadly, which overlaps heavily with semantic conflicts but is not exclusively semantic-only. Detection: only behavioural tests covering the interaction surface catch these (arXiv 2310.02395 — unit-test generation best-but-imperfect across 51 scenarios). High for the concept.

3) Test each merge against the actual landing state (merge queues).

The stale-base problem

A, B, C all branch from example-branch@X; each passes CI against X; A merges (now X+A); B’s CI never saw A; B merges untested-against-A; a semantic A–B conflict lands silently. Fix: test each candidate against base + all already-queued changes.

Critical caveat High

Merge queues only catch what the tests cover — they enforce that coverage runs against the right target state; they do not replace behavioural tests. GitHub / GitLab / Graphite primary.

Worktrees — what they do and don’t give High

A worktree is a separate working directory sharing the same object DB and refs (per-worktree HEAD + index + files; shared commits + branch pointers). They give filesystem isolation (work 3 branches without stash / switch). They give zero merge safety — when one worktree merges into example-branch, all others see it immediately via the shared ref store; nothing protects against semantic conflicts. Git docs even warn multi-checkout is “still experimental”. Worktrees are a productivity tool; the integration discipline above is separate and required regardless.

The solo case — “it’s just me” is wrong with 3 worktrees

The “other developers” harmed by a broken example-branch are your future self in the other two worktrees.

MSP-driven target
  • Plan tasks as MSPs (add the non-breaking criterion + flag/keystone/expand-contract guidance + dependency-graph gate).
  • Validate each task / wave against baseBranch before proceeding (close the mid-run-breakage gap); merge-queue-style landing-state test for parallel tasks.
  • Require tests covering the interaction surface where tasks share state.
Counter-case — objectivity
  • Decomposition into independently-mergeable MSPs costs upfront design time; a tightly-coupled feature (payment flow across UI / API / schema / 3rd-party) may be cheaper to do serially.
  • Flag overhead compounds: 3 flags = 8 paths; QA test-space explosion; toggle debt (Fowler “inventory with a carrying cost”; acidtango counter-view).
  • Some tasks genuinely can’t parallelize (direct dependency) — map the dependency graph first; same-level tasks parallelize, dependent tasks serialize or use expand-contract.
  • Merge queues add latency under heavy load (minor at 3 tasks; real at 50+ PRs/day).

05 Clarification 4 — WIP noise, branch isolation, squash-on-merge define + the squash/rebase/merge trade-offs

Glossary
TermPlain meaning
WIP commita quick “work in progress” checkpoint save (WIP / fix typo / oops); for crash-safety + local rollback, not meant to be clean
WIP noisemany WIP commits cluttering shared history; hard to read, breaks bisect (many aren’t buildable)
Branch isolationWIP commits stay on a private / feature branch; the shared branch only ever sees the cleaned-up result
Squash-on-mergecollapse all of a branch’s commits into ONE commit at merge time
Merge commita 2-parent commit joining histories (git merge --no-ff); keeps all branch commits; non-linear history
Rebase-and-mergereplay each branch commit onto the base (new SHAs); linear; keeps granularity
git bisectbinary search across commits to find the one that introduced a bug; needs testable commits
The tension + how it resolves

You want frequent small saves (crash-safety, agent checkpoints, easy rollback) and a clean atomic published history. They live in different scopes. Branch isolation keeps WIP on the branch; squash-on-merge converts the noisy branch into one clean entry on the shared branch. The branch is the scratchpad; the shared branch is the published record.

WIP → squash — two scopes, one clean published commit
feature branch (private):  [WIP]->[WIP]->[fix]->[WIP]->[checkpoint]
                                                              | squash
shared branch (published):   ... [one clean atomic commit] <-+

The glossed-over downside of squashing High

Squash collapses N commits into 1, so for bisect that PR is a single node — if the bug was in the 3rd of 12 squashed commits, bisect can only point at the PR, not the commit. Mitchell Hashimoto: “Squashing destroys this information.” So squash is right for messy WIP, wrong when each intermediate commit is itself atomic and meaningful. The question before squashing: “would any of these individual commits be a meaningful bisect target?”

Tie to MSP — the right squash granularity
GranularityResult
one squash per giant multi-MSP featureone enormous commit; bisect near-useless; blunt revert
one squash per MSP (one PR per MSP)one atomic clean commit per shippable unit; bisect finds the MSP; surgical revert; clean log
no squash (rebase-and-merge) when commits already atomic+greenfull sub-MSP granularity; needs commit-hygiene enforcement
Pragmatic default for agent workflows Med-High

Agent commits during execution are WIP by nature, so the safe default is one squash per MSP PR with a clean message — eliminates WIP noise, gives MSP-level bisect + surgical revert + readable log. Use rebase-and-merge only if intermediate commits are genuinely atomic + green (rare without enforcement).
