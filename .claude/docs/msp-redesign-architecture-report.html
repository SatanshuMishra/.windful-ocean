MSP-driven parallel development · teaching report

Determining and Running Minimum Shippable Products, end to end

A complete re-design of how a written spec becomes parallel, independently-mergeable slices of work that never break the shared branch. Written for a smart reader with minimal domain knowledge: every term is defined in plain words on first use, and everything comparative is a table, diagram, or callout rather than a wall of prose.

Section 01 · Answer first

BLUF / TL;DR

Six messages carry the whole design. Everything after this section is detail and evidence.

01

MSPs come from a 7-step algorithm, not vibes

Spec → capability backbone → candidate vertical slices → INVEST + non-breaking gate → dependency DAG from 5 signals → topological sort → clusters. The same pure topological-sort the current engine already ships (planWaves) does the layering — run twice (once over MSPs, once over tasks). The hard part — deciding hidden dependencies the spec never states — MUST stay human-gated.

02

What runs it: small additions wrapping the old chain

THREE new skills + ONE new agent + TWO new engine modules, wrapping (not discarding) today's brainstorming → writing-plans → annotation → execute chain. The old chain becomes the INNER loop, now run once PER MSP instead of once per feature.

03

The orchestrator-of-orchestrators is native

Claude Code subagents can spawn subagents (depth-5 cap, since v2.1.172). main(0) → per-MSP orchestrator(1) → task agents(2) sits at depth 2 with 3 levels of headroom. Native isolation: worktree handles per-feature/per-cluster worktrees.

04

The merge flow IS a merge queue

"Branch X never breaks" is TRUE for structural breakage IF merges are serialized + tested against the current tip + the CI gate cannot be bypassed. It does NOT prevent semantic conflicts (two MSPs each green alone, broken together) — only interaction-surface tests can, and that residual is an honest, irreducible gap.

05

Graphify computes the regression surface, with 3 gaps

graphify affected "X" --depth N (verified on live data) gives the blast radius, but has 3 load-bearing gaps needing supplements: label ambiguity (pair with Serena), no parallel-surface command (build a small wrapper), no test-coverage map (use coverage data or an import heuristic).

06

Confidence: HIGH on capabilities and theory

HIGH on capabilities (verified against live tooling + official docs) and on the merge-queue and decomposition theory (primary sources). The two places the vision depends on something outside pure automation: the human cluster-plan gate, and interaction-surface test coverage.

Section 02 · Plain-words anchor

Glossary

Read this once and the rest reads easily. Each term is defined the way a patient senior engineer would explain it at a whiteboard.

Feature
One whole thing the user asked to build. One brainstorming session → one SPEC document.
SPEC
The written design output of brainstorming; the input to the new pipeline.
MSP (Minimum Shippable Product)
A vertical slice of a feature that can merge to the shared branch on its own WITHOUT breaking the running app, and delivers observable value (or safely-latent code). Equivalent to "thin vertical slice" / "independently releasable story" in industry literature.
Vertical slice
A narrow strip of work that goes through ALL the layers it needs (UI → API → DB) for one behavior — like a slice of layer cake — instead of building one whole layer at a time (a "horizontal slice"). Vertical slices are independently shippable; horizontal ones are not.
Walking skeleton
The very first, thinnest vertical slice that proves the whole architecture connects end to end (Alistair Cockburn). The first MSP of any project.
Cluster
A group of MSPs that MUST run in a set order because they depend on each other. Independent MSPs go in SEPARATE clusters so the clusters can run in parallel.
Task
One bite-sized implementation step inside a single MSP's plan.
Branch X
The shared integration branch every finished MSP lands on. The invariant the whole design protects: Branch X is always green (passing tests).
Worktree (git)
A second (or third) working copy of the same repository, each on its own branch, fully separate on disk but sharing one git history. Lets parallel work happen without file collisions.
Graphify
A local code-knowledge-graph CLI. Given a changed symbol, it returns everything that depends on it (its "blast radius"). Used to choose which tests to re-run after a change.
Node / Edge / Community
A node is one symbol (function, file, constant); an edge is a relationship ("A calls B", "A imports B"); a community is a cluster of related nodes Graphify groups together.
Dependent
Code that USES (calls/imports/references) another piece of code.
Regression surface
The set of code (and the tests covering it) that COULD break because of a change. After pulling in others' changes, this set can grow.
Parallel surfaces
Code that references the SAME symbol as your change without calling your code — siblings that move together (e.g., two callers of the same constant).
Merge queue
A system that serializes merges so each change is tested against the CURRENT tip of the target branch before it lands, keeping the branch always-green (Bors / GitHub merge queue / GitLab merge trains).
Semantic conflict
Two changes that git merges with NO textual conflict, but whose combined BEHAVIOR is wrong. Git is a text differ; it knows nothing about meaning.
Test Impact Analysis (TIA)
Mapping "which code changed" → "which tests could fail" and running only that subset.
Expand-and-Contract (parallel change)
A 3-phase way to change a shared thing safely — add the new alongside the old (Expand), migrate callers one at a time (Migrate), remove the old only when nothing references it (Contract).
Subagent
A worker Claude spawns via the Agent tool, with its own fresh context window; returns only a summary to its parent.
Orchestrator
An agent whose job is coordination (split work, dispatch workers, collect results), not doing the work itself.

The correctness claim, stated precisely

IF the dependency graph is COMPLETE and each MSP's non-breaking technique is correctly applied, THEN individually-correct MSPs compose into a correct feature once merged. The "if" is exactly why Step 4 and the technique-tagging are human-gated.

Section 05 · Concern 2 answered

What actually runs this

Today there is NO skill that takes a spec, finds MSPs, and plans each. Here is exactly what we add, and how it reuses what already exists.

5.1 · The new pieces

New skill

feature-decomposition

Trigger = after brainstorming approves a SPEC, on "decompose / break into MSPs". Input: SPEC path. Output: <feature>.clusters.json + markdown cluster plan. Runs the 7-step algorithm, calls planWaves at MSP granularity, computes clusters, STOPS at the human gate. NEW front segment between brainstorming and planning.

New skill

msp-planning (thin wrapper)

Invoked by an MSP orchestrator for ONE MSP. Decides if that MSP needs its own scoped brainstorming; invokes writing-plans on ONLY that MSP's slice; runs parallel-plan-annotation → msp.graph.json. This is the precise sense in which the OLD chain becomes the INNER loop — run per MSP, not per feature.

New skill

feature-orchestration

After the cluster plan is approved, opens the Feature worktree, launches cluster worktrees, dispatches one msp-orchestrator per MSP respecting cluster order, collects summaries, drives the merge queue.

New agent

msp-orchestrator (depth 1 or 2)

Tools include Agent (MUST — to spawn task agents), Read, Bash, Serena, graphify, the skills. isolation: worktree on the cluster branch. Per MSP: msp-planning → parallel-subagent-development (existing engine, scoped to this MSP) → relays review/test/security → returns a concise summary + "ready to merge". RESTRICT spawnable types via disallowedTools, NEVER a parenthesized type-allowlist (ignored at subagent level — confirmed in Claude Code docs).

New engine/lib

msp-merge-protocol.mjs (+ a merge lock)

The serialized merge queue + regression-surface builder (Section 7).

New engine/lib

graphify-regression.mjs

The gap supplements (community wrapper + Serena bridge + coverage map), phased.

6.3 · Level-collapse rule

When only ONE feature is active (the default), the main thread IS the Feature orchestrator — the natively-supported main(0) → MSP(1) → task(2) shape. Only spawn a separate Feature-orchestrator layer when ≥2 features run concurrently. The depth-5 cap is never approached.

Each layer returns a BOUNDED structured summary, never raw transcripts. Task agents return the existing one-line STATUS: DONE — ≤50 words. MSP orchestrator returns {mspId, status, mergedSha|blockReason, filesTouched, reviewVerdicts}. Feature orchestrator returns {feature, mspsMerged, mspsBlocked, branchXSha}.

Section 07 · Step by atomic step

The MSP merge protocol

This is a merge queue — a serialized gate where each change is tested against the CURRENT tip before it lands.

7.1 · Why both ingredients are mandatory

Pull-before-merge (test against current tip) AND a non-bypassable gate. Neither alone suffices. Pull-without-gate = one "force merge" breaks everyone. Gate-without-pull = green against a 3-day-old tip is meaningless.

GitHub merge queue "ensure[s] the pull request's changes pass all required status checks when applied to the latest version of the target branch" (GitHub docs); the underlying principle is the "not rocket science rule" (Bors / Graydon Hoare).

7.2 · The merge-queue landscape

SystemMechanismSafety vs throughput
Bors / not-rocket-sciencepure serialization, test each PR on current tipmax safety, low throughput (~32/day at 15-min CI)
GitHub merge queuemerge-group branches tested on latest tip; required checks tied to merge_group event (non-bypassable)balanced; serial or grouped
GitLab merge trainsparallel pipelines, each assumes prior cars landfaster; a failure restarts downstream cars
Graphitestack-aware; serialized OR batched w/ binary-search bisecttunable
Aviatorserial/batched/optimistic/optimistic-batched spectrum; can skip re-validation for independent paths via build graphtunable

Sources: GitLab trains · Graphite batching · Aviator · Earthly merge queues

The MSP worktree is already green at L3 (tasks integrated, per-MSP boundary validation passed, review + security passed). Only THEN does it enter the queue.

7.5 · The semantic-conflict gap

Gap · honest limitation

"Individually green" is necessary but NOT sufficient. Two MSPs each green alone can break combined while git merges cleanly. The structure can FORCE the relevant interaction-surface tests to run; it cannot guarantee they were written. This is Risk R1.

Worked example — a multi-currency change that silently overcharges. Read the code top to bottom.

Branch X at T0
remove_money(account_id, amount) {
  deduct(account_id, amount)
}
MSP-Alpha adds multi-currency — updates all KNOWN call sites, tests pass, merges. X now at T1.
remove_money(account_id, amount, currency='USD') {
  deduct(account_id, convert(amount, currency))
}
MSP-Beta (developed from T0) adds a new payment flow calling the OLD 2-arg form. Tests pass against T0.
remove_money(account_id, amount)

Beta pulls X (T1). Git merge: CLEAN (no textual overlap). The new signature accepts currency with default 'USD'; Beta's call uses the default.

Production: Beta's flow charges EVERY customer in USD regardless of their currency — yuan charged at ~7x. Branch X is broken; tests didn't catch it.

Why no merge queue catches this: a merge queue tests structure, not meaning. ONLY a test that exercises the interaction surface catches it. Step 2's whole purpose is to FORCE interaction-surface tests into the expanded set by graph-traversing to new callers. Residual risk remains wherever no such test exists — an honest, irreducible gap (Risk R1).

Sources: Semantic Conflict · currency example

A pricing_api_test.js that asserts response.body.pineapple_price === 1.99 does NOT import the constant — a pure call-graph misses it. Catch via (a) a coverage map for prices.js, or (b) a literal search for the old value "1.99" across tests.

8.2 · The three load-bearing gaps + supplements

Gap 1 · Label ambiguity (highest severity)

affected/path/explain match by label string, not unique ID. 39% of labels are non-unique (main()=41 nodes, SKILL.md=288). affected "main()" unions 41 reverse traversals = useless for common names.

Supplement: Serena find_referencing_symbols by file+symbol for exact callers. Graphify finds the neighborhood; Serena pins the precise callers.

Gap 2 · No community / parallel-surface command

Community is a per-node integer but no command enumerates it. Supplement: a small jq/python wrapper over graph.json grouping by community.

Gap 3 · No test-coverage map

Graphify knows build.js calls ensureDir() but not that utils.test.js exercises it. Supplement: an external coverage map (lcov/Istanbul) or an import-graph heuristic.

Also note

Semantic/INFERRED edges are DISABLED (AST-only; all semantic_hash empty); shell-script call graphs are shallow.

8.3 · Per-commit regeneration wiring

So "many small atomic commits" keeps the graph fresh.

Feasible + incremental

A manifest tracks per-file mtime + ast_hash; only changed files re-extract. Measured: ~0.3-1s for a few files, ~6.8s full rebuild; "no topology change" leaves graph.json untouched.

Git projects

Install graphify's native post-commit + post-checkout hooks (graphify hook install), WRAPPED to run backgrounded (nohup) and DEBOUNCED (skip if an update launched < ~3s ago) so rapid commit bursts don't thrash. Post-commit becomes the primary freshness driver during active work.

Reconcile with existing hooks

SessionStart provision = cold-start build-if-absent; Stop refresh = backstop. The find -newer guard + debounce dedupe naturally (if post-commit already updated, Stop finds nothing newer → no-op).

Per-worktree graphs

Each worktree reflects its own state; the merge protocol runs graphify on the MSP worktree AFTER the pull, so the blast radius reflects post-pull reality.

8.4 · Freshness-lag safety rail

The graph reflects the LAST update; shell call graphs are shallow; semantic edges disabled. So for files edited since the last update (the just-pulled diff and the in-flight wave), verify exact caller facts with Serena/LSP, NOT the possibly-stale map. This is diff-local — re-verify only the recent diff, not the whole graph. This rail is exactly why merge step 2c UNIONS graphify (breadth, maybe stale) with Serena (precise, live).

Wiring file refs: hooks/graphify-provision.sh, hooks/graphify-refresh.sh, graphify-out/graph.json (+ manifest.json), settings.json hook config.

Section 11 · The user's decisions

Key design forks

Each fork below is OPEN — it needs your decision. The architect's pick is marked RECOMMENDED.

Fork 1

Orchestration mechanism

A. All live nested subagents (no Workflow engine) — max parallelism, but re-derives the engine's deterministic merge/review logic in prose (fragile).
B. Workflow engine everywhere — battle-tested + ceiling-immune, but its one-level nesting can't express multi-MSP parallelism.
C. HYBRIDRecommended
Nested subagents for Feature/cluster/MSP orchestration; the Workflow engine for the within-MSP task waves. Each tool in its proven range.
Fork 2

Worktree granularity & MSP merge target

A. Worktree per MSP, MSP→X directly — perfect isolation, many worktrees (disk/ports/node_modules xN).
B. One Feature worktree, MSPs scope-fenced inside — cheap, but kills cross-cluster parallelism + intra-tree collision risk.
C. Worktree per CLUSTER, MSP→X directlyRecommended
Worktree count = number of independent clusters (the real parallelism unit); sequential MSPs in a cluster reuse one worktree (no intra-cluster race); each MSP still merges to X individually. Merge target = MSP→Branch X directly, Feature branch fast-forwarded to X after each landing.
Fork 3

Decomposition authority

A. Auto-decompose unattended — fast, but 12% failure silently breaks X (unacceptable).
B. Fully human-authored MSP plan — safe, wastes the LLM's reliable 80%.
C. LLM-proposes / human-approvesRecommended
LLM does the reliable 80%; the mandatory cluster-plan gate covers hidden coupling, technique, value, cycles.
Fork 4

Merge-queue concurrency

A. Strictly serializedRecommended for v1
One MSP at a time under a global lock; deterministic; trivial to reason about; MSP volume per feature is low so throughput is a non-issue.
B. Optimistic (parallel speculative merges, retry) — higher throughput, adds speculation+rollback machinery not needed yet.
C. Batched trains — amortizes CI, but red-batch bisecting is complex.
Fork 5

How much graphify-gap tooling to build now

A. Build all three supplements now — most complete, but the lcov coverage-map pipeline is the heaviest lift and delays everything.
B. Serena exact-caller bridge now; defer the restRecommended
Closes the highest-severity gap (label-ambiguity) immediately; approximate parallel surfaces via affected --depth; import-graph heuristic for coverage until a real lcov map is justified.
C. Build none, use raw affected — unsafe (affected "main()" unions 41 traversals).

Section 13 · Plain truth

Two honest caveats

The architecture cannot STRUCTURALLY eliminate two things; both must be stated plainly.

Caveat 1 · The semantic-conflict gap (R1)

Structure can only FORCE the relevant interaction-surface tests to exist and run; it cannot guarantee they were written. "Branch X never breaks" is precisely: "Branch X will not break in any way the test suite covers, given serialized + tested-against-tip + non-bypassable merges."

Caveat 2 · The human cluster-plan gate (R6)

Hidden coupling the spec doesn't state needs a human who knows the codebase. This is the one mandatory human checkpoint.
