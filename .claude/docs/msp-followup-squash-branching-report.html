Per-MSP Squash, and Targeting a Branch Without Guessing Its Name

A focused answer to two pointed objections raised against the prior MSP report: (1) does per-MSP squash throw away the point of atomic commits, and (2) how does industry reliably identify the branch to target when names are unenforced. Evidence-led, code-grounded, and willing to say “your instinct is correct.” Read-only deliverable — no configuration was changed.

00 Bottom line up front two questions, two answers

Q1 · Genuine tradeoff
“Why per-MSP squash? Why not leave the hundreds of atomic commits in place? Isn’t squashing losing the point of atomic commits?”
Squash does not destroy atomicity — it scopes it to two granularities. Atomic commits serve the developer and reviewer on the feature branch (work-in-progress navigation, granular review, mid-PR debugging). On the protected branch the PR/MSP itself becomes the atomic unit, and squash enforces one-logical-change-per-commit there. The real cost is losing sub-PR git bisect granularity in the published history — bounded, because the feature branch is retained for archaeology. Grounding: our engine never squashes today (it merges --no-ff), so this would be a new behavior, not a fix. A strong alternative keeps both linear history and per-commit granularity: rebase-and-merge with small/stacked PRs.
Verdict: present a 3-way choice. Do not silently adopt squash.
Q2 · Confirmed
“Detect-then-confirm is still flawed — you cannot assume dev/development is the integration branch, main can be named anything, and nothing enforces standards. Why not declare the strategy, or pass the exact target?”
Your instinct is correct and the evidence is unambiguous. Heuristic name detection is not the industry standard — the one tool that hardcodes branch names (semantic-release) had to be publicly patched when the industry moved master → main, proving the fragility. Every well-designed tool either requires the caller to state the base, derives it from the repo’s API-configured default branch, or reads stored machine-readable config. Both alternatives you proposed are validated by practice. And our engine already implements the most robust path — baseBranch is a required CLI parameter with zero name-guessing.
Verdict: kill detect-then-confirm. Replace with declare-or-pass, never guess.

02 Concern #1: per-MSP squash vs preserving atomic commits a genuine tradeoff

An atomic commit is one small, self-contained, logical change that leaves the tree in a working state. The skeptic view (squash throws away atomicity) and the integration view (the PR is the atomic unit) are both right — at different levels.

2.1 — The premise shift: one word, two granularities
Feature-branch level
atomic = one testable change within the feature
audience: developer + reviewer

Work-in-progress navigation, granular review, mid-PR bisect/debugging. The hundreds of atomic commits do their job here — and the feature branch is retained, so they are never lost.

Integration-branch level
atomic = one deployable, reviewed PR/MSP
audience: history, releases, reverts

One PR → one commit → one changelog entry → one-command revert. Squash enforces “one logical change per commit” on the protected branch — exactly the rule you already apply to published history.

2.3 / 2.4 — Why teams squash, and why they don’t
For squash
  • Google release-please recommends it: “We highly recommend that you use squash-merges” — cleaner history, easier bisect, changelog control; feature-branch messages “don’t make sense when merged in the main branch.” github.com/googleapis/release-please
  • Conventional Commits supports it explicitly — maintainers clean up the message at merge. conventionalcommits.org/v1.0.0
  • Bisect reliability (not just cleanliness): every squashed commit on main is a reviewed, CI-passing state, so git bisect never lands on a broken WIP commit. danielrotter.at
  • One-command whole-feature revert: git revert <squash-commit>.
  • “Commit often, perfect later, squash on merge” is a first-class documented workflow: squash “gives your team the freedom to commit as often as they want, without… unnecessary commits to the main repository.” GitLab: squash and merge
Against squash
  • Linux kernel: every patch is already a standalone atomic unit that builds and runs — “Separate each logical change into a separate patch.” The patch is the squash equivalent; they never bundle. kernel.org: submitting-patches
  • Mitchell Hashimoto prefers merge commits ~90% of the time; squashing a big diff destroys bisect value and loses authorship. gist: mitchellh
  • GitHub’s own docs flag drawbacks: loses authorship; git rerere is less effective; long-lived branch reuse causes previously-squashed commits to reappear in later PR diffs. (same GitHub merge-methods doc)

2.7 — For / Against / Alternative
For squash · High confidence

Clean integration history; reliable PR-granularity bisect; trivial whole-feature revert; developer freedom (“commit often, perfect later”); release-tool fit (release-please recommends it outright).

Against squash · High confidence

Permanent loss of sub-PR bisect granularity in published history; lost authorship; bundles separable changes; weaker rerere; semantic-release needs extra config (CC-formatted PR titles); long-lived-branch reuse hazard (squashed commits reappear).

Alternative that dissolves the tension entirely

Rebase-and-merge + small / stacked PRs. Keeps linear history and per-commit granularity if every commit is atomic and CI-passing — removing the squash-vs-preserve tension. graphite.com: PR merge strategy. Lower-ceremony variant: interactive-rebase cleanup before opening the PR, then merge-commit or rebase-merge.

2.8 — Verdict for the lever (decision-ready)

The squash lever is a genuine tradeoff, not an obvious win, and we don’t squash today. If MSP = “one PR = one shippable unit,” squash maps cleanly (1 MSP → 1 commit → 1 changelog entry → 1-command revert) — that is the case for the lever. The cost (sub-PR bisect) is real but bounded by keeping MSPs small, which the MSP discipline already pushes. Recommendation: present a 3-way choice — keep --no-ff / squash-per-MSP (with CC-formatted PR title) / rebase + small PRs. Do not silently adopt squash.

03 Concern #2: how industry reliably targets a branch no guessing

3.1 — The flaw, confirmed

Name-based detection — assume dev/development/staging is the pre-main integration branch, assume main/master is the mainline — is not the industry standard. The one prominent tool that hardcodes names is semantic-release (default list ['master','main','next','beta','alpha', …]). It had to be publicly patched to add main when the industry migrated master → main — concrete proof of fragility. issue #1581 · config docs

Every other primary tool avoids name-guessing: GitHub REST makes base required; GitLab REST makes target_branch required; the gh CLI falls back to the API-configured default branch (not a name scan); Renovate and Dependabot read the API default branch.

3.2 — Ranked robust mechanisms (most → least deterministic)
Centerpiece · branch-target mechanism ranking
RankMechanismDeterministic?Where the truth livesFailure modeTooling
1 Explicit per-PR base (caller declares) Yes — no inference The caller’s invocation Only if the caller is wrong gh pr create --base; GitHub REST base required (pulls API); GitLab target_branch required (merge_requests API)
2 Repo’s configured default branch Yes for single-mainline Repo settings (admin UI / API) Wrong for git-flow (default = main but integration = develop) GitHub default_branch field + GITHUB_DEFAULT_BRANCH; gh repo set-default; GitLab default_branch; git symbolic-ref refs/remotes/origin/HEAD (local cache, can be stale)
3 Stored branching config Yes — machine-readable .git/config / committed .gitflow Absent unless git-flow initialized git config --get gitflow.branch.develop (git-flow config reference)
4 Branch-protection / ruleset enumeration Partial — over-inclusive Repo ruleset / protection config Multiple protected branches; “important” ≠ “the PR target” GitHub GET /branches?protected=true, Rulesets API
5 Name-based heuristic No — convention-dependent Unenforced naming conventions Any non-conventional name; multi-branch models; renames semantic-release defaults; ad-hoc CI if ref == 'main' scripts

High Rank 1 (explicit base) is exactly what our engine already does — and what GitHub’s REST API mandates.

3.3 — Your two proposals, each validated

(a) Explicit per-PR target

“PR from worktree → branch X”, “branch X → development”. This is Rank 1, the most robust mechanism, and exactly what GitHub’s REST API mandates. It is also what our engine already does.

Verdict · Adopt as the default contract — every MSP run states its base; never inferred.

(b) Declare the strategy per project

In config / CLAUDE.md. Valid and with direct precedent — git-flow stores gitflow.branch.develop; semantic-release stores a branches array. But a critical refinement applies (see 3.4).

Verdict · Valid — but the declaration must be machine-readable + verified, not prose alone.

(c) Enforce a default on fresh projects

Standard practice. git flow init -d (creates master+develop and writes machine-readable config in one command); GitHub template repos (carry committed config); git init -b main / init.defaultBranch; org default-branch policy; cookiecutter post-gen hooks. git-init · GitHub org default branch

Verdict · Standard — enforce via real setup commands, not prose.
3.4 — The refinement that makes “declare” robust
The drift trap · the single most important nuance

Declaring the model in CLAUDE.md prose carries high drift risk. Nothing validates that “we use gitflow, main+develop” in CLAUDE.md matches the actual branches. Under your own no-comments.md philosophy (“the code is the only reliable source of truth; comments drift”) and memory-discipline.md (“if memory and code disagree, code wins”), a prose branch declaration is a hint, not authority.

Industry has no cross-tool schema for “branch roles” — each tool invents its own (git-flow INI keys; semantic-release branches array; changesets a single baseBranch). changesets config-file-options

Therefore the robust shape is layered — authority lives in the machine-readable + live git state, with CLAUDE.md as a verified label only.

1
Authoritative = machine-readable + live git state

The actual branches; a committed config key if a tool needs one. This is what automation trusts.

2
CLAUDE.md = human / AI-readable label + pointer

“This project uses trunk-based; base = main.” The agent must verify it against the real branch before acting — never the sole source.

3
Fresh projects = enforce a default via real commands

A setup step (e.g. git flow init -d, git init -b main), not prose.

Net: your “declare in CLAUDE.md” instinct is right for intent, but the agent must treat it as a hint and confirm against reality — exactly the discipline already encoded in this config.

3.5 — Recommended revised lever: declare-or-pass, never guess

A three-tier resolution order for the base branch — all deterministic, none heuristic:

1Explicit pass winsRank 1

If the run/PR states --base X (or “worktree → X”), use X.

2Else read a declared machine-readable sourceRank 2–3

A committed branch config key, or the repo’s API / HEAD default branch.

3Else stop and ask the human

Our engine already has the “on main/master, stop and ask” guard — generalize it to “if base is neither passed nor declared, ask; never infer from a name.”

×Never

Scan branch names for dev / develop / staging / main and assume a role.
